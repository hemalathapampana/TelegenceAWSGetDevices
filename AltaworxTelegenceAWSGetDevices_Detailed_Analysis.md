## 1. SQS Event Triggers

### 1.1 What generates the SQS event?

The SQS events that trigger this Lambda function are generated by **the Lambda function itself** through a self-enqueuing pattern. This creates a continuous processing loop for handling large datasets.

**Trigger Sources:**
- **Self-enqueuing**: The Lambda re-enqueues itself to continue processing when there's more data
- **Manual/Scheduled invocation**: Initial trigger can be manual or from a scheduled job
- **Other Lambda functions**: Can be triggered by other parts of the Telegence sync pipeline

**Key SQS Message Generation Points:**
```csharp
// Lines 223, 229, 503, 683, 765 in AltaworxTelegenceAWSGetDevices.cs
await SendMessageToGetDevicesQueueAsync(context, syncState, TelegenceDestinationQueueGetDevicesURL, delaySeconds);
```

### 1.2 SQS Message Attributes Structure

The SQS messages contain detailed state information:

| Attribute | Type | Purpose |
|-----------|------|---------|
| `CurrentPage` | String | Tracks pagination through Telegence API |
| `HasMoreData` | String (bool) | Indicates if more API data is available |
| `CurrentServiceProviderId` | String | Identifies which service provider is being processed |
| `InitializeProcessing` | String (bool) | Determines if BAN status initialization is needed |
| `IsProcessDeviceNotExistsStaging` | String | Flags device validation phase |
| `GroupNumber` | String | Groups devices for batch processing |
| `IsLastProcessDeviceNotExistsStaging` | String | Marks final validation group |
| `RetryNumber` | String | Tracks retry attempts for timeout handling |

### 1.3 Processing Flow Triggers

1. **Initial Run**: `InitializeProcessing = true`, processes BAN statuses first
2. **Continuation Run**: `InitializeProcessing = false`, processes device data
3. **Validation Run**: `IsProcessDeviceNotExistsStaging = true`, validates missing devices
4. **Retry Run**: `RetryNumber > 0`, handles timeout/error recovery

## 2. SQL Retry Initialization

### 2.1 Why SQL retry is implemented as the first step

SQL retry is implemented immediately upon Lambda initialization (line 183) to protect against **database connectivity issues** and **transient failures** that are common in cloud environments.

```csharp
// Line 183: AltaworxTelegenceAWSGetDevices.cs

```

### 2.2 Issues it specifically protects against

**Primary Protection:**
- **Connection timeouts** during database initialization
- **Transient network failures** between Lambda and SQL Server
- **Database deadlocks** during high-concurrency operations
- **Connection pool exhaustion** in shared database environments

**Implementation Details:**
- Uses **Polly library** for resilient database operations
- Applies to ALL database operations throughout the Lambda execution
- Configured with `CommonConstants.NUMBER_OF_RETRIES` attempts
- Implements **exponential backoff** strategy

**Specific Usage Points:**
```csharp

    // Database operation with automatic retry
```

### 2.3 Retry Configuration

- **SQL Operations**: `CommonConstants.NUMBER_OF_RETRIES` (typically 3-5 attempts)
- **API Operations**: `CommonConstants.NUMBER_OF_TELEGENCE_RETRIES` (for Telegence API calls)
- **Lambda Retries**: `CommonConstants.NUMBER_OF_TELEGENCE_LAMBDA_RETRIES` (for SQS re-enqueuing)

## 8. Error Handling & Retry Mechanisms

### 8.1 Retry Configuration Details

**Polly Retry Policies:**

1. **SQL Operations**: 
   - **Policy**: `policyFactory.SqlRetryPolicy(CommonConstants.NUMBER_OF_RETRIES)`
   - **Attempts**: Typically 3-5 retries
   - **Strategy**: Exponential backoff

2. **API Operations**:
   - **Policy**: `PollyRetryForProxyRequestAsync(context.logger, CommonConstants.NUMBER_OF_TELEGENCE_RETRIES)`
   - **Attempts**: Configured for Telegence API reliability
   - **Strategy**: Exponential backoff with jitter

3. **Lambda Re-enqueuing**:
   - **Condition**: `syncState.RetryNumber <= CommonConstants.NUMBER_OF_TELEGENCE_LAMBDA_RETRIES`
   - **Increment**: `syncState.RetryNumber++` on timeout
